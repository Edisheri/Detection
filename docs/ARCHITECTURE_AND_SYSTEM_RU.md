# LungDx Pro — архитектура ПО и разбор кода

Документ для защиты, видео и самопроверки: **что за система**, **из чего состоит**, **какая нейросеть**, **какой файл за что отвечает**.

---

## 1. Назначение системы

**LungDx Pro** — локальное программное обеспечение на **Python** с веб-интерфейсом (**Streamlit**). По загруженной **рентгенограмме грудной клетки** выполняется:

1. Проверка «похож ли файл на рентген» (эвристика, **без нейросети**).
2. Классификация **нейросетью ResNet18** на **5 классов**: `COVID-19`, `Cancer`, `Normal`, `Pneumonia`, `Tuberculosis`.
3. Вывод вероятностей, уровня риска, текстовых рекомендаций, при необходимости **Grad-CAM**, экспорт **PDF/JSON**, история и очередь исследований.

Система **не заменяет врача**; это поддержка принятия решений / исследовательско-учебный комплекс.

## 2. Логическая архитектура (слои)

```
┌─────────────────────────────────────────────────────────────────┐
│  Представление (UI)          Streamlit: app.py                  │
│  • формы, загрузка файлов, графики, PDF, session_state        │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│  Прикладная логика             app.py (функции-хелперы)         │
│  • пороги, приоритеты, SLA, риск, дифф. диагностика, отчёты     │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│  ML / качество входа          src/inference.py                  │
│  • load_model, predict_image, generate_gradcam                  │
│  • is_likely_chest_xray (фильтр до модели)                     │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│  Модель                        src/model.py                     │
│  • torchvision ResNet18 + головка классификации (5 классов)    │
└────────────────────────────┬────────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────────┐
│  Данweights: weights/best_model.pt ← train.py + датасет         │
└─────────────────────────────────────────────────────────────────┘

Обучение и оценка (офлайн): train.py, scripts/evaluate.py, src/metrics.py
Конфигурация: config.py
```

**Где «работает нейросеть»:**

- При **обучении**: в процессе `train.py` на **GPU, если доступна CUDA**, иначе CPU.
- В **веб-приложении** (`run_app.py` → `app.py`): при вызове `predict_image` / Grad-CAMPyTorch загружает веса и считает на **CPU** (в `load_model` устройство задаётся CPU — это сознательный обход падений Streamlit на Windows с некоторыми сборками CUDA).

## 3. Нейросеть: что это и из чего состоит

### 3.1 Архитектура

Реализация: **`src/model.py`**, функция **`build_model`**:

| Компонент | Описание |
|-----------|----------|
| **База** | `torchvision.models.resnet18` с весами **ImageNet** (`IMAGENET1K_V1`) при обучении с `pretrained=True` |
| **Замена головы** | Вместо исходного `fc` — последовательность `Dropout(0.3)` → `Linear(in_features → num_classes)` |
| **Выход** | **Логиты** размерности `num_classes` (у вас **5**). **Softmax** не внутри `nn.Module`: при обучении его роль берёт на себя `CrossEntropyLoss`; при инференсе — `torch.softmax` в **`predict_image`** |

Класс **`LungDiseaseClassifier`** — обёртка-наследник `nn.Module`; фактически в проекте чаще используется результат `build_model()` напрямую.

### 3.2 Что сохраняется после обучения

Файл **`weights/best_model.pt`** (checkpoint PyTorch):

- `model_state_dict` — все параметры ResNet18 + головки
- `num_classes`, `class_names`, `image_size`

Дополнительно:

- **`weights/class_names.json`** — список имён классов (удобно без чтения checkpoint)
- **`weights/training_history.json`** — списки `train_loss`, `train_acc`, `val_loss`, `val_acc`, `best_acc`

### 3.3 Как учится модель (train.py)

Краткий алгоритм **`train.py`**:

1. Читает **`TRAIN_DIR` / `VAL_DIR`** из **`config.py`** (папки с подкаталогами по классам).
2. Строит **`ChestXRayDataset`** + **`DataLoader`** (`src/dataset.py`).
3. Создаёт модель **`build_model(..., pretrained=True)`**, переносит на `cuda` или `cpu`.
4. **Потери**: `CrossEntropyLoss` с **весами классов** (`compute_class_weights`) — борьба с дисбалансом.
5. **Оптимизатор**: `Adam` с **разными learning rate** для backbone и для головки (`fc`): backbone `lr * 0.1`, голова `lr`.
6. **Планировщик**: `CosineAnnealingLR(T_max=число эпох, eta_min=1e-6)`.
7. Цикл эпох: `train_epoch` → `validate`; лучший checkpoint по **val accuracy** пишется в **`best_model.pt`**.
8. В конце: полная оценка через **`evaluate_model`** (`src/metrics.py`), отчёты в **`reports/`**.

Число эпох по умолчанию берётся из **`config.NUM_EPOCHS`**; для эксперимента с 20 эпохами запускают `train.py --epochs 20`.

### 3.4 Как вызывается модель при анализе снимка

**`src/inference.py` → `predict_image`**:

1. Опционально можно вызывать **`is_likely_chest_xray`** до инференса (в **`app.py`** фильтр выполняется **до** `predict_image`, чтобы не тратить ресурс на «левые» картинки).
2. Изображение приводится к тензору через **`get_transforms(image_size, train=False)`**: resize 224×224, нормализация ImageNet.
3. `model(x)` → логиты → `softmax` → словарь вероятностей, класс, уверенность.
4. В словарь добавляются **`xray_score`** и детали фильтра (для отображения предупреждений в UI).

**Grad-CAM** (`generate_gradcam`): хуки на последний блок **`model.layer4`**; обратное распространение от выбранного класса; карта важности масштабируется на исходный размер, **без OpenCV** (PIL + NumPy).

## 4. Фильтр «это рентген?» — не нейросеть

**`is_likely_chest_xray`** в **`src/inference.py`** — классическая **эвристика** по пикселям (резкая картинка уменьшается до 256×256 для скорости):

Основные идеи:

- **Попиксельная серость** (`per_pixel_diff`): настоящий рентген в RGB часто с `R≈G≈B`; цветные/тонированные фото дают больший разброс каналов.
- **Средняя насыщенность** в псевдо-HSV смысле: у рентгена низкая «цветность».
- **Симметрия** левой и отражённой правой половины (типичный ПА/антеропостериорный грудной снимок).
- Вспомогательно: контраст, соотношение сторон, доля тёмного фона.

Итог: флаги **`is_xray`**, **`score`** (0…1) и **`details`** для вывода в интерфейсе при отказе.

## 5. Датасет и предобработка

**Файл: `src/dataset.py`**

- **`ChestXRayDataset`**: обходит подпапки классов, собирает пары `(путь_к_файлу, индекс_класса)`.
- **`get_transforms`**:
  - **train=True**: resize с запасом, random crop 224, flip, поворот, ColorJitter, лёгкий affine, ToTensor, Normalize ImageNet.
  - **train=False**: детерминированный resize 224×224, ToTensor, Normalize.

Пути к данным задаются в **`config.py`**: **`CHEST_XRAY_DIR`** выбирается из кандидатов (`data/chest_xray` или `data/data/chest_xray`), далее **`TRAIN_DIR`**, **`VAL_DIR`**.

## 6. Метрики и отчёты

**Файл: `src/metrics.py`**

- **`evaluate_model`**: прогон `DataLoader`, softmax по логитам, сбор `y_true` / `y_pred`, расчёт **confusion_matrix**, **classification_report** (sklearn), accuracy, macro/weighted precision, recall, F1.
- Сохранение: тексты/JSON, PNG матрицы ошибок, **`metrics_summary.json`** (может включать историю обучения при вызове из `train.py`).

Папка **`reports/`** — артефакты оценки; для графиков см. **`scripts/generate_metrics_visual.py`**.

## 7. Веб-приложение: структура app.py

**Файл: `app.py`** (монолит UI + часть бизнес-логики; функции, не ООП-классы Streamlit).

| Элемент | Назначение |
|--------|------------|
| **`get_model`** (`@st.cache_resource`) | Однократная загрузка **`load_model(WEIGHTS_PATH)`** |
| **`_load_metrics_summary` / `_load_training_history`** | Чтение **`reports/metrics_summary.json`**, **`weights/training_history.json`** для Model Card |
| **`_quality_assessment`** | Простая оценка яркости/контраста/«резкости» для предупреждений (отдельно от `is_likely_chest_xray`) |
| **`_risk_level`, `_decision_status`, `_confidence_band`** | Клиническая подача: риск, авто-вердикт, полоса уверенности |
| **`_recommendations`** | Текстовые рекомендации по предсказанному классу |
| **`_priority_*`, `_sla_*`, `_enqueue_worklist`, `_worklist_view_rows`** | Очередь: приоритеты STAT/Срочно/Планово, SLA, таблица рабочего листа |
| **`_build_case_pdf`, `_build_session_pdf`** | Сборка PDF по кейсу и по смене (reportlab и т.п. внутри файла) |
| **`_render_result`** | Главный блок результата: дифф. диагностика Cancer/Pneumonia, прогресс-бары вероятностей, кнопки скачивания JSON, интеграция Grad-CAM |
| **`main`** | Sidebar: роль, порог, quality gate, приоритет, Model Card; основная колонка: загрузка, проверка X-ray, инференс, история, вкладки аудита |

**Сессия:** `st.session_state` хранит историю анализов, аудит-лог, хэш последнего события (чтобы не дублировать строки).

**Запуск:** **`run_app.py`** выставляет переменные окружения (**отключение CUDA** до импорта torch), затем `streamlit run app.py` на порту **8501**.

## 10. Поток данных «от файла до PDF» (для устного объяснения)

1. Пользователь загружает файл в Streamlit.
2. Приложение строит `PIL.Image`, при необходимости считает **`_quality_assessment`**.
3. **`is_likely_chest_xray(image)`**: если **не рентген** → сообщение с метриками, **`st.stop()`**, **модель не вызывается**.
4. **`predict_image(model, ...)`** → класс, вероятности, `xray_score` для предупреждения о качестве.
5. **`_render_result`** формирует вердикт, рекомендации, экспорт; при низкой уверенности — предупреждения.
6. Опционально **`generate_gradcam`** → тепловая карта.
7. История/аудит пополняется; PDF собирается из payload.

## 13. Возможные вопросы комиссии — короткие ответы

**Почему ResNet18, а не ViT / YOLO?**  
Задача — **классификация всего снимка** по фиксированным классам; ResNet18 компактен, хорошо работает с transfer learning и медицинскими датасетами.

**Где граница между «правилами» и «нейросетью»?**  
Отсев не-рентгена — **правила** (`is_likely_chest_xray`). Класс болезни — **нейросеть**.

**Почему CPU в интерфейсе?**  
Стабильность **Streamlit на Windows** при известных конфликтах CUDA; обучение при этом может идти на GPU отдельным процессом.
